Network Traffic Analysis Using Wireshark

Project Overview

This project involved a forensic investigation of the "Catbomber" malware incident. Using Wireshark, I analyzed network traffic to identify compromised systems, spot command-and-control traffic, and pull out evidence of the attack.

Key Findings

• Identified two infected clients (10.5.28.229 and 10.5.28.8) through HTTP traffic filtering and TLS handshake analysis.

• Pulled out exfiltrated credentials including usernames, hostnames, email addresses, and passwords from captured packets.

• Verified malicious files as Indicators of Compromise using VirusTotal.

• Detected fake SSL certificates used in command-and-control communications.

Screenshots & Analysis

VirusTotal Confirmation
File hash submitted to VirusTotal showing that many vendors flag cursor.png as malicious, confirming it as an Indicator of Compromise.
Email Credentials Exfiltration
HTTP stream showing Outlook credentials being sent in clear text from the infected client to the attacker’s server.
Second Infected Client
IP configuration output used to confirm the hostname and IP of the second compromised machine.
Primary Infected Client
Network details of the first compromised workstation, giving context for filters and follow-up analysis.
Malware Extraction
Using Wireshark’s Export HTTP Objects feature to pull out suspicious files for offline analysis.

Technologies Used

Wireshark • PCAP Analysis • Network Security Monitoring • HTTP/TLS Analysis • VirusTotal • PowerShell

Password Authentication & Hash Cracking

Project Overview

This project looks at how Windows NTLMv2 authentication works on the wire and why weak passwords are a bad idea. I captured NTLMv2 logons in Wireshark, exported the pieces I needed, and used Hashcat on Kali to try cracking them with a common wordlist.

Workflow

• Capture NTLMv2 logons over the network using Wireshark filters.
• Pull out the server challenge and NTLMv2 response from the authentication packets.
• Build a hashes.txt file in the format Hashcat expects and move it over to Kali.
• Run Hashcat (mode 5600) with the rockyou.txt wordlist to test password strength.

Screenshots & Analysis

Capturing NTLMv2 Response
Filtering on ntlmssp to find the NTLMv2 authentication packet and copying the response and NTProofStr values from it.
Server Challenge Value
From the server challenge packet, I grab the 8-byte challenge that, combined with the response, becomes the input for Hashcat’s NTLMv2 mode.
Building hashes.txt
On the Windows side, I paste together username, domain, challenge, and response into user::domain:challenge:response format and save it as hashes.txt.
Moving Hashes to Kali
Browsing to the Windows host from Kali and downloading hashes.txt over HTTP, so all hash cracking happens offline.
Hashcat Attack Command
Running Hashcat in NTLMv2 mode (-m 5600) against hashes.txt using the rockyou.txt wordlist to simulate a basic offline attack.
Cracking Progress & Stats
Hashcat status output, showing speed, progress, and keyspace. Any weak passwords would show up here as recovered.
Authentication Context
NTLM details in Wireshark that tie each captured hash back to a real user, domain, and workstation on the network.

Key Takeaways

• NTLMv2 hashes can be captured by anyone with access to internal traffic.
• Once hashes are offline, attackers can try as many passwords as they want without lockouts.
• Long, unique passwords plus MFA make this kind of attack much harder to pull off.
