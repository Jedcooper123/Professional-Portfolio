Network Traffic Analysis Using Wireshark

Project Overview

In this lab I worked through the “Catbomber” scenario, which simulates a malware infection inside a company network. I used Wireshark and a PCAP file to figure out which machines were infected, what data was stolen, and which files were actually malicious.

How I Approached It

• Started by opening the PCAP and looking at high-level HTTP and TLS traffic.
• Followed specific TCP streams to see what data was being sent out from clients.
• Used IP configuration data in the packets to tie activity back to hostnames and users.
• Exported suspicious files from the capture, hashed them, and checked them on VirusTotal.
• Looked at post-infection TLS traffic and certificates to spot command-and-control behavior.

Screenshots & Analysis

Primary Infected Client
Following one of the HTTP streams shows a system report that includes the hostname Cat-Bomb-W7-PC with IP address 10.5.28.229. This becomes our first confirmed infected workstation.
Second Infected Client
Another HTTP stream shows a different host, Catbomber-DC, with IP 10.5.28.8. The usernames match the first machine, which suggests the same user is compromised on multiple devices.
Stolen Email Credentials
By following another HTTP stream from 10.5.28.229, I see the user’s Outlook email address and password being sent to the attacker over clear-text HTTP. This confirms that credentials are being exfiltrated, not just system info.
Pulling Malware from the PCAP
Using File → Export Objects → HTTP in Wireshark, I extract suspicious files like cursor.png and save them into a quarantine folder so they can be analyzed without touching the live network.
Hashing & Checking the File
In PowerShell I generate a file hash for the extracted PNG and submit it to VirusTotal. This lets me check quickly whether security vendors have already seen this file in the wild.
Confirmed Indicators of Compromise
VirusTotal shows that 61 out of 72 vendors flag cursor.png as a trojan, which confirms it as a strong Indicator of Compromise for this environment.
Suspicious Post-Infection Traffic
Looking at TLS sessions on ports 443, 447, and 449, I find a certificate whose commonName is simply example.com. Combined with the other evidence, this points to malicious command-and-control traffic rather than normal web use.

Key Takeaways

By walking through the PCAP step by step, I was able to identify two infected hosts, see exactly what data left the network, and confirm which files were malware. This is the kind of workflow you’d use in a real incident: use traffic to rebuild what happened instead of guessing.

Intrusion Detection & Prevention System

Project Overview

This lab focuses on Snort running on a pfSense firewall. I set up custom and pre-built rules, watched alerts fire in real time, and then used packet captures and Wireshark to prove what the alerts were actually seeing on the wire.

How I Approached It

• Enabled Snort on the LAN interface of pfSense with a custom.rules file for testing.
• Wrote a simple rule that alerts any time traffic includes my name “Jed” going to or from the Windows server, just to prove rules are working.
• Hosted a suspicious PNG file on a Kali Apache server and had the Windows client download it.
• Captured traffic on pfSense, pulled it into Wireshark, and used an Emerging Threats rule set to detect the PNG that secretly contains an EXE.
• Verified inside Wireshark that the “image” payload actually starts with an MZ executable header.

Screenshots & Analysis

Custom Snort Rule
Creating a test rule in custom.rules that looks for the string “jed” in any TCP traffic and raises a friendly alert. It’s a simple way to confirm Snort is inspecting the traffic I care about.
Rule Firing in the Log
Snort’s Active Log shows alerts generated by that custom rule when traffic containing “Jed” passes between 192.168.1.2 and the Windows server at 10.1.0.1. This confirms that my rule is live and matching the expected flows.
Hosting the Malicious File
On Kali, I start Apache and host cursor.png. From the user’s point of view it’s just a download, but in the background Snort sees it as suspicious content moving through the firewall.
HTTP Download in Wireshark
Filtering in Wireshark to show the HTTP GET request for /cursor.png and the server’s 200 OK response. This ties the Snort alert to an actual file transfer.
Packet Capture from pfSense
Packet capture output confirms the TCP sessions between the Kali web server and the internal host. These packets are what Snort is analyzing when it decides to raise an alert.
Network Trojan Detected
An Emerging Threats rule fires with the message “Content-Type image containing Portable Executable data.” In other words, Snort has figured out that this “PNG” is actually carrying a Windows executable.
Exporting the Payload
Using Wireshark’s Export Objects feature to pull the suspicious file out of the traffic so it can be handed off to a sandbox or malware analyst.
PE Header Hidden in a PNG
Following the TCP stream shows the HTTP response claiming to be image/png, but the payload begins with the MZ header and PE sections. That’s a clear sign that the file is actually an executable wrapped in image clothing.

Key Takeaways

This lab shows how IDS/IPS tools like Snort fit into a layered defense. Rules give you early warning, but the real value comes from validating those alerts with packet captures and understanding the story behind each event.

Password Authentication & Hash Cracking

Project Overview

This project looks at how Windows NTLMv2 logins travel over the network and what happens if an attacker captures those logins. I used Wireshark to grab NTLMv2 authentication traffic, built a hash file in the format Hashcat expects, and then tried cracking it with a common password list.

How I Approached It

• Captured NTLMv2 authentication traffic on the network using Wireshark filters.
• Pulled out the server challenge and NTLMv2 response fields from the packets.
• Built a hashes.txt file in the user::domain:challenge:response format that Hashcat needs.
• Moved the hash file over to Kali Linux and ran Hashcat (mode 5600) with the rockyou.txt wordlist.
• Reviewed the results to see whether any of the users were using weak, crackable passwords.

Screenshots & Analysis

Finding the NTLMv2 Response
Filtering on ntlmssp in Wireshark highlights the NTLMv2 authentication packet. From here I copy the NTLM response fields that will later be fed into Hashcat.
Server Challenge
The server challenge packet provides the 8-byte random value that, combined with the user’s password and other fields, forms the NTLMv2 hash. Attackers need both pieces: the challenge and the response.
Building hashes.txt
On the Windows side, I paste together username, domain, challenge, and response into user::domain:challenge:response format and save it as hashes.txt. This is exactly what Hashcat expects when cracking NTLMv2 hashes.
Moving Hashes to Kali
From Kali, I browse to the Windows system and download hashes.txt over HTTP. From this point on, all cracking happens offline so there are no account lockouts or logs on the domain controller.
Hashcat Attack Command
Running Hashcat in NTLMv2 mode (-m 5600) against hashes.txt with the rockyou.txt wordlist. This simulates what a basic attacker might do after stealing hashes from a network.
Cracking Progress
Hashcat status output shows speed, percentage complete, and how many hashes (if any) have been recovered. Weak passwords would show up here quickly.
Tying Hashes Back to Users
Looking back at the NTLM details in Wireshark, I can match each hash to a real domain user and workstation. That’s important for reporting and for recommending which accounts need password resets.

Key Takeaways

NTLMv2 hashes are not plain-text passwords, but once they’re captured, they can still be attacked offline. Long, unique passwords and multi-factor authentication make this kind of attack much harder, which is why those controls matter even on internal networks.
